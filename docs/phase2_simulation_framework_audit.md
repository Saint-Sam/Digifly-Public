# Phase 2 Simulation Framework Audit

Date: 2026-04-22

Sources reviewed:

- Phase 2 simulation framework
- Hemilineage Simulations workspace

## Bottom Line

The reusable Phase 2 foundation is already mostly concentrated in `Phase 2/digifly/phase2`. The Hemilineage Simulations folder is valuable, but it should not be imported wholesale. It is roughly 22 GB and mostly contains generated project folders, cached runs, paper-specific DNg100 experiments, plots, videos, and notebook launch state.

The first public Phase 2 pass should use this shape:

- keep the Phase 2 Python package
- keep the NEURON mechanism source files
- keep focused tests
- keep one canonical user-facing notebook
- document the Hemilineage/DNg100 experiments as future examples, not framework core

That is now staged in `Digifly Public/Phase 2`.

## Common Ground Across Phase 2 Simulations

### Configuration

Core modules:

- `digifly.phase2.config.defaults`
- `digifly.phase2.config.loader`
- `digifly.phase2.config.schema`
- `digifly.phase2.neuron_build.config`

These define the shared runtime knobs: selected neuron IDs, SWC roots, edge paths, run roots, timing, synaptic defaults, biophysics, gap junctions, CoreNEURON, CVODE, recording policy, cache controls, and parallel build settings.

Best public role: keep as framework core.

### SWC And Synapse Data Loading

Core modules:

- `digifly.phase2.data.paths`
- `digifly.phase2.data.synapses_loader`
- `digifly.phase2.data.master_table`
- `digifly.phase2.neuron_build.swc_cell`
- `digifly.phase2.neuron_build.ais`

These resolve SWC files, synapse CSVs, metadata tables, export indices, AIS sites, and morphology-backed NEURON sections.

Best public role: keep as framework core. Public docs should say that large SWC/synapse exports are generated by Phase 1 and are not committed.

### Edge And Circuit Construction

Core modules:

- `digifly.phase2.graph.requested_edge_sets`
- `digifly.phase2.graph.edge_cache`
- `digifly.phase2.graph.edges_from_synapses`
- `digifly.phase2.graph.edges_loader`
- `digifly.phase2.graph.custom_circuit_workflow`
- `digifly.phase2.graph.filters`
- `digifly.phase2.graph.nodesets`

These provide the common edge schema, named edge-set registry, dataset-aware edge signatures, one-hop expansion, edge-cache SQLite build/query, and custom circuit overlays.

Best public role: keep as framework core. This is one of the strongest shared areas between the hemilineage notebooks, DNg100 notebooks, and custom circuit notebooks.

### NEURON Build Layer

Core modules:

- `digifly.phase2.neuron_build.network`
- `digifly.phase2.neuron_build.builders`
- `digifly.phase2.neuron_build.biophys`
- `digifly.phase2.neuron_build.gaps`
- `digifly.phase2.neuron_build.ownership`
- `digifly.phase2.neuron_build.parallel`
- `digifly.phase2.neuron_build.wiring_plan`
- `digifly.phase2.neuron_build.timing`
- `digifly.phase2.neuron_build.site_mapping`
- `digifly.phase2.neuron_build.viz_ais`

These are the actual reusable simulation primitives: load cells, map synapses to morphology, apply passive or HH biophysics, wire chemical synapses, compile/load gap mechanisms, build serial or distributed networks, map external neuron IDs to compact NEURON gids, run CoreNEURON where possible, and save outputs.

Best public role: keep as framework core.

Cleanup note: `walking/runner.py` still has several debug prints in the run path. They do not block the staged copy, but they should become optional logging before a polished release.

### Runners And Outputs

Core modules:

- `digifly.phase2.walking.runner`
- `digifly.phase2.hemi.runner`
- `digifly.phase2.hemi.sim_project`
- `digifly.phase2.util.save`
- `digifly.phase2.util.logging`
- `digifly.phase2.util.progress`

These turn configs into runs and write outputs such as config JSON, voltage records, spike records, phase timings, cell biophysics summaries, edge metadata, run summaries, benchmark tables, and optimization summaries.

Best public role: keep as framework core, but split `hemi/sim_project.py` later. It is almost 3,000 lines and currently mixes project layout, edge building, run orchestration, reduction sweeps, coalescing sweeps, benchmark planning, plotting, and summary analysis.

### Tests

Current useful tests:

- `test_requested_edge_sets.py`
- `test_parallel_build_planning.py`
- `test_distributed_save_merge.py`
- `test_distributed_empty_rank_recording.py`
- `test_repo_structure.py`

These protect the parts that matter for a reusable framework: edge-set registration, Phase 1 fallback plumbing, export index fallback, parallel ownership, distributed record/spike merging, CoreNEURON fallback, and repo structure.

Best public role: keep and expand.

## Hemilineage Simulations Triage

### Keep As Future Public CLI Tools

These are useful, but should be adapted before importing because their path assumptions still point at the old master workspace layout:

- `run_hemi_quick_project.py`: launches `run_full_hemilineage_project()` from a JSON case payload, with optional serial preflight and MPI support.
- `run_hemi_cached_session.py`: long-lived hemilineage cache server that builds once and serves repeated runs.
- `run_explicit_cached_session.py`: long-lived explicit-config cache server for repeated custom network runs.
- `run_parity_case.py`: compact single-case launcher around `run_walking_simulation()`.
- `hemi_setup_validator.py`: useful setup validation candidate.

Recommended public destination later:

```text
Phase 2/scripts/
  run_hemi_project.py
  run_cached_hemi_session.py
  run_cached_explicit_session.py
  run_phase2_case.py
```
